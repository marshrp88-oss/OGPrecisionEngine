RESERVE — April 30 Master Improvement Plan
Audience: Replit Agent 4, executed on April 30 when credits restore.
Authority: This document supersedes both FIX_PLAN.md and FIX_PLAN_PART_K.md for the April 30 session. Where any of them conflict, this one wins.
Scope: All currently-known defects, missing features, data corrections, and editability gaps, ordered by priority and engineering relevance.
Meta-principle for this session: Every value displayed in the UI must be editable from that UI. The user should never need to prompt the AI agent to change their own data. If there is a number on screen that the user owns (balance, bill amount, commission, snapshot, score, parameter), there must be a click-to-edit affordance within 1 click of that number. This is not a feature — it is a design principle that must hold across every page.

P0-1. Discretionary This Cycle — new primary Dashboard output
Category: Missing core output (the most important missing thing in the app)
User Impact: This is the number the user built the entire system to produce. Safe to Spend answers "will I miss a bill" (defensive). Monthly Savings answers "what will this month leave behind" (retrospective). Neither answers the daily decision-making question: "what money is truly free — for savings, investing, or wants, without downstream consequence?" Without this, the app fails its core use case.
Engineering Impact:

New function discretionaryThisCycle(ctx) in lib/finance/financeEngine.ts
New card on Dashboard, co-equal with Safe to Spend
Unit test required

Formula:
typescriptfunction discretionaryThisCycle(ctx: FinanceContext): number {
  const safeToSpend = computeSafeToSpend(ctx);
  
  // 1. Variable spend remaining at target pace through next payday
  const daysToPayday = daysUntil(ctx.nextEffectivePayday);
  const variableRemaining = Math.max(0, 
    (daysToPayday / ctx.monthLengthDays) * ctx.variableSpendCap
  );
  
  // 2. Cushion for bills in NEXT cycle (between next payday and payday+15)
  // that the next paycheck can't cleanly absorb at $0 commission
  const nextCycleBills = ctx.bills
    .filter(b => b.include === true)
    .filter(b => isDueInNextCycle(b, ctx.nextEffectivePayday))
    .reduce((s, b) => s + b.amount, 0);
  const nextPaycheckNet = ctx.baseNetPerPaycheck; // ~1575 at base
  const nextCycleVariable = (15 / ctx.monthLengthDays) * ctx.variableSpendCap;
  const nextCycleShortfall = Math.max(0, 
    nextCycleBills + nextCycleVariable - nextPaycheckNet
  );
  
  // 3. Buffer for irregulars (QuickSilver accrual not yet logged + 10% contingency)
  const quicksilverUnlogged = ctx.variableSpendLog.currentMonthAccrual || 0;
  const contingency = safeToSpend * 0.10;
  
  const discretionary = safeToSpend 
    - variableRemaining 
    - nextCycleShortfall 
    - quicksilverUnlogged 
    - contingency;
  
  return Math.max(0, discretionary);
}

P0-5. Bills Page "In Cycle Hold" Badge Logic Fix
Category: Bug in existing feature
User Impact: The Bills page shows five bills flagged "In cycle hold" for the current cycle when the correct answer is zero (everything is post-paid for the April 22 cycle, and the next cycle anchored at May 7 is not yet current). The user cannot trust the badges.
Engineering Impact: Small surgical fix. Likely a wrong payday reference in the Bills page component.
Root cause hypothesis: The Bills page is likely computing "in cycle" using nextPayday = today + 15 days or similar heuristic instead of pulling the live nextEffectivePayday from the shared FinanceContext used by the Dashboard.
Fix:

Audit the cycle-hold badge computation in the Bills page component
Ensure it uses the same nextEffectivePayday that safeToSpend uses
Ensure the strict < rule from BUILD_SPEC.md §4.4 is applied (bills due ON payday are excluded from current cycle)

Verification:

On April 21 with payday April 22: only bills with next due date strictly between today and April 22 are flagged. Result: zero badges.
Advance system clock to April 23: bills with next due date strictly between April 23 and May 7 are flagged. Result: five badges (Car Loan May 1, Verizon May 2, Claude May 3, Rent May 4, EZ-Pass May 22 — wait, EZ-Pass is day 22 which equals May 7? No, May 22 > May 7, so excluded).
Actual five-in-cycle for a post-April-22 state: Car Loan (5/1) + Verizon (5/2) + Claude (5/3) + Rent (5/4) = 4 bills, not 5. Confirm EZ-Pass is correctly excluded until May 22 is in a cycle.
