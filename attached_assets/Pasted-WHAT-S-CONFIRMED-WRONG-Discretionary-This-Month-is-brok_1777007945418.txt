Three defects identified after code audit. Fix in order.
DEFECT 1 — Forward Reserve double-counts bills already in the current cycle hold.
Current behavior: forwardReserve() sums all Include=TRUE bills with due_day between 1-7 regardless of whether they're already captured in the current cycle's Required Hold. When today is before a payday and a bill's next_due_date falls between today and payday (making it part of Required Hold), that same bill with due_day 1-7 is ALSO in the Forward Reserve. This causes double-counting in Monthly Savings Estimate and Discretionary This Month calculations.
Example: Today April 24, next payday May 7. Car Loan (day 1) has next_due_date May 1, which is in the May 1-7 window. It appears in Required Hold ($1,423 total) AND in Forward Reserve ($1,561.16 total).
Correct behavior: Forward Reserve should represent bills due in the 1-7 window of the NEXT month AFTER the upcoming payday. If a bill in the 1-7 window is already being reserved via Required Hold for the current cycle, it should not be counted again.
Fix: modify forwardReserve() to accept currentCycleBills: Bill[] parameter representing bills already in the current cycle's Required Hold. Exclude those from the 1-7 sum. Update all callers of forwardReserve() to pass the appropriate list. Update both Python reference and TypeScript engine.
